## Issues (should fix before next user-facing launch)

### 1. Save as Home/School is silent — Nielsen #1 (visibility of system status)

The place card's "Save as Home" and "Save as School" buttons write to
localStorage with no confirmation feedback. The place card stays
identical. Verified: after clicking "Save as Home" on Brandenburg Gate,
`localStorage.bike-route-home` was populated, but the UI gave no
indication that anything had happened.

**Impact:** a reasonable user will tap it twice, or think the button
broke.

**Fix:** minimal — swap the button label to "✓ Saved as Home" for 2–3
seconds after tapping, then revert (or keep as "Saved" state if the
current place IS the saved Home, using a lat/lng comparison). Even
simpler: briefly flash a check mark inline.

### 2. Mode picker moves location between states

**Desktop:** top-right in W1/W2/W3/W5, but the routing state (W4) moves
the picker to the bottom-left of the route summary card (see code:
`src/App.tsx:726` — the picker is rendered inside the route summary
card during routing; the top-right mount is hidden via
`.ui-routing .map-travel-mode { display:none }` implicitly via the
conditional wrap).

**Impact:** users learn two locations for the same control. A user
mid-route trying to switch from kid-starting-out to kid-confident to
compare routes has to hunt for the picker.

**Fix:** pick one. The route-summary-card placement is actually the
better one (it's context-aware: mode affects the route you're looking
at). Remove the top-right mount when `uiState === 'routing'` would
make the current behavior intentional; remove the in-panel mount if
you want a single stable location.

### 3. Saved Home/School label is opaque

The quick-option reads just "🏠 Home" once saved, with no hint of the
actual address. A user who saved the wrong place (or who has multiple
devices) can't tell from the quick-option label whether their saved
Home is actually what they think it is. Only discovery is tapping the
button, getting an unexpected route, and realizing.

**Fix:** include the saved `shortLabel` as a subtitle on the quick
option: `🏠 Home / Dresdener Straße 112`. Supports recognition over
recall (Nielsen #6).

### 4. "Rough" marker on route is unexplained

The route shows a segment-type marker labelled "Rough" floating mid-
route, where the path crosses a cobblestone section. A family-cycling
novice has no idea whether "Rough" is a warning or a surface type.
No tooltip on hover.

**Fix:** the marker already signals something meaningful
(surface=cobblestone / kid bike handling). Add a 1-line descriptor
under the label: `Rough surface — walk the bike here`. Or change the
marker to an icon (⚠️ or 🚷) that reads faster.

### 5. Legend move-arrows (↓) still unlabelled

The legend row for "Bike path" has a `↓` button that moves the item to
"Other" (so it stops being green on the map). Today's work increased
the tap target to 32×32, but the affordance is still opaque — the
arrow looks like it would collapse the row, not reclassify the path
type.

**Fix:** either replace the icon with something more semantic (maybe a
`–` or `×` when in Preferred, `+` when in Other), or add tooltip text
that shows on long hover / first render. The interaction is powerful
but hidden.

### 6. Status + attribution leak through the admin overlay

When the admin panel opens (position: fixed, inset: 0), the Leaflet
zoom controls (bottom-right) and attribution link are visible behind
the panel because the panel is `background: #fff` but doesn't
explicitly cover those controls. Small visual cruft.

**Fix:** either raise `.audit-overlay` z-index above leaflet (currently
900; leaflet default is 1000+), or hide the leaflet controls while
admin is open with `body.admin-open .leaflet-control { display: none }`.

## Polish (post-ship OK)

### 7. Mode picker icons lack persistent labels

The five mode icons are visually distinctive but not self-explanatory
without hover tooltips. On first visit, a user has to hover each to
learn what each means. On mobile (where hover doesn't exist), they
have to tap and see.

**Low-priority fix:** add a tiny label under each icon at desktop
widths (≥768px). Or surface the mode name in a one-line banner above
the picker that updates on hover.

### 8. Profile-picker "OTHER" heading with eyeball

The legend "OTHER" section collapses by default and reveals an eyeball
icon to toggle other-path visibility. Two interactions (collapse chevron
vs eyeball) on the same line is a minor learnability issue but feels
like expected map-UI behaviour.

### 9. No onboarding guidance for mode picker meaning

A new user sees 5 small bike icons in the top-right. They have to
guess which icon is which mode. Consider a one-time tooltip or
side-panel explainer on first visit ("Pick the rider profile that
matches who you're routing for").

### 10. Route summary card uses "X" to dismiss → goes all the way back

Tapping the × on the route summary doesn't just hide the panel —
it exits routing mode entirely and returns to the search state. A
user who wants to tweak settings without closing the route loses the
route. This is standard in some apps but worth signalling.

## What worked well

- **Cold-land clarity.** The overlay + mode picker + search bar lets a first-time user understand "this is a bike-infrastructure map" in under 3 seconds.
- **Empty-state guidance** on Home/School ("Tap to add …") and on the Samples tab ("Scan a city first"). Neither dead-ends.
- **URL state is honest.** `?travelMode=...&admin=...` means any state is shareable and back-button-safe. This is unusually good for a React SPA and helps discoverability / recovery.
- **Classified route polyline.** The two-tone green / orange segmentation is the product value proposition, and it lands immediately.
- **Quality bar** (65% preferred / 35% other) gives an at-a-glance honest signal. Users can calibrate.
- **Autocomplete quality.** Structured-address queries return 1–2 clean Nominatim results for typical Berlin queries, not 20 noisy matches.

## Review limitations

1. **Mobile viewport could not be forced.** macOS Chrome window minimum
   exceeds 375px; `resize_window` and `window.resizeTo()` both report
   success but `innerWidth` stays at 1440. Mobile findings here are
   CSS-derived + based on today's touch-target fixes. A real-device or
   Chrome DevTools device-emulation pass is recommended for a proper
   mobile UX review.
2. **Admin "Scan" flow not exercised.** Would take several minutes to
   scan Berlin; W5 was only tested through the empty-state.
3. **Review did not cover recovery flows** (network error, geolocation
   denied, geocoding fail). All worth a dedicated pass.

## Summary

**Ship-block count: 0.** Every core workflow completes on desktop.

**Issues worth fixing before you push the app to a wider audience:**
- Save-as-home silence (#1) is the sharpest — one user session with a
  stranger and they'll tap Save twice.
- Mode-picker relocation (#2) is a learnability tax worth paying once to fix.
- Saved-Home opacity (#3) hits every user who saves a Home and comes back a week later.

**Polish queue:** mode labels (#7), onboarding for mode meaning (#9).
